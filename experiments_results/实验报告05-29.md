# 实验报告 05-29

## 一、baseline3 实验

### 1. baseline3 的目标

baseline1 和 baseline2 更偏向直接让 LLM 生成或根据 WP 反馈重试。baseline3 的核心变化是：不再只把失败 WP 日志整体丢给 LLM，而是把失败证明目标拆出来，做“失败原因诊断 + 局部 assertion 探测+ 定向 repair”。

baseline3 的目标可以概括为：

```text
LLM 初始生成 ACSL
  -> Frama-C/WP 验证
  -> 若失败，解析 failed proof obligation
  -> 诊断失败类别
  -> 对 proof_gap / missing_spec / ambiguous 先尝试 assertion probe
  -> 根据 probe evidence 或诊断结果做 repair
  -> 再运行 WP 验证
```

不是单纯要求 LLM 重新生成完整规格，而是尝试把失败点定位到某个 proof obligation，再围绕这个 proof obligation 做更小粒度的修复。

### 2. baseline3 的主要流程

baseline3 的入口是：

```text
scripts/run_baseline3.py
```

主要流程如下：

```text
1. 初始生成
   使用 baseline1 的 direct generation prompt 先生成一版 ACSL。

2. WP 验证
   运行 Frama-C/WP，使用 Alt-Ergo 和 Z3，默认 WP timeout 为 8 秒。

3. 失败目标抽取
   extract_failed_proof_obligations() 从 WP 日志中解析失败的 proof obligation。
   记录字段包括 name、status、line_no、goal_text、assumptions、prove_statement 等。

4. 失败上下文抽取
   extract_wp_goal_context() 从详细 Goal block 中提取 Assume {...} 和 Prove: ...。

5. 失败原因诊断
   diagnose_failed_po() 把失败归类为：
   - wrong_spec
   - missing_spec
   - proof_gap
   - ambiguous

6. assertion探测
   对 proof_gap / missing_spec / ambiguous，baseline3 会先插入局部 probe marker：
   - loop 前：PROBE_HERE:loop*_before
   - loop body 入口：PROBE_HERE:loop*_body_entry

   然后让 LLM 生成若干局部 ACSL assertion：
   //@ assert <predicate>;

   这些 assertion 不是最终 invariant，而是用来测试“这个中间事实是否能帮助 WP 证明当前目标”。

7. repair proposal
   根据失败类别选择修复 prompt：
   - wrong_spec：rewrite/delete 可疑 annotation
   - missing_spec：添加 invariant、contract、assigns 等持久规格
   - proof_gap：优先尝试局部 assertion，必要时 promote 成 invariant/contract
   - invalid：修复语法或非法 ACSL

8. 安全检查
   validate_generated_code() 会拒绝以下情况：
   - LLM 修改可执行 C 代码
   - LLM 删除或改写原始目标 assertion
   - original_text 无法唯一匹配

9. 多轮修复
   每个 case 默认最多 3 次初始生成，每次失败后最多 7 轮 repair。
```

baseline3 的关键设计是把 LLM 输出限制在更小的动作空间内：先诊断，再生成 assertion candidate，再把有用的局部事实提升成规格，而不是让 LLM 无约束地重写整个文件。

### 5. baseline3 的有效性分析

baseline3 相比 Autospec 的主要优势不是模型更强，而是流程更结构化：

```text
Autospec:
  主要依赖 LLM 直接生成规格或整体修复。

baseline3:
  把 WP 失败拆成 proof obligation，
  对失败目标做分类，
  用 assertion probe 测试中间事实，
  再做定向 repair。
```

这种流程带来几个好处：

1. 失败信息更精确  
   LLM 不需要从完整 WP 日志中自行定位失败点，而是直接看到 `assumptions` 和 `prove_statement`。

2. repair 动作更受控  
   wrong_spec、missing_spec、proof_gap 对应不同 prompt，减少了无关修改。

3. assertion 探测提供实验性证据  
   局部 assertion 可以判断某个中间事实是否真的帮助当前 VC，而不是直接把猜测写成 invariant。

4. 代码安全检查降低破坏性  
   baseline3 拒绝修改可执行 C 代码、删除原始 assertion 等行为，避免 LLM “修证明但改程序”。

#### 6.1 原始程序和初始规格

原始程序的核心逻辑是：

```c
i = 1;
j = 10;

while (j >= i) {
    i = i + 2;
    j = j - 1;
}

//@ assert(j == 6);
```

循环变量的实际变化过程大致是：

```text
(i, j) = (1, 10)
       -> (3, 9)
       -> (5, 8)
       -> (7, 7)
       -> (9, 6)
```

然后循环条件 `j >= i` 变成 `6 >= 9`，循环退出。因此最终确实应该有：

```text
j == 6
```

baseline3 初始生成的规格是：

```c
/*@
  loop invariant j >= i ==> (i + j == 11);
  loop invariant j < i ==> (j == 6 && i == 11);
  loop assigns i, j;
*/
```

这两个 invariant 都有问题。

第一个：

```text
j >= i ==> i + j == 11
```

只在初始状态 `(1,10)` 成立。循环一次后状态变成 `(3,9)`，此时 `j >= i` 仍然成立，但：

```text
i + j = 12
```

所以它不是 inductive invariant。

第二个：

```text
j < i ==> (j == 6 && i == 11)
```

也不正确。真实退出状态是：

```text
i == 9
j == 6
```

而不是 `i == 11`。因此初始规格既有非归纳的问题，也有事实错误的问题。

#### 6.2 trace 中每一轮发生了什么

本次 trace 共有 6 轮 WP 检查，前 5 轮失败，第 6 轮成功：

| 轮次 | WP 结果 | 失败点 | baseline3 动作 |
| --- | ---: | --- | --- |
| round 00 | `Fail_6_8` | invariant preservation | 重写错误 invariant |
| round 01 | `Fail_5_8` | invariant preservation + assertion | 改成核心线性关系 |
| round 02 | `Fail_5_6` | assertion `j == 6` | 添加基础上下界 |
| round 03 | `Fail_13_14` | assertion `j == 6` | 添加桥接 invariant |
| round 04 | `Fail_15_16` | 新 invariant preservation | 弱化过强 invariant |
| round 05 | `Pass_16_16` | 无 | 全部通过 |

##### round 00：识别初始 invariant 非归纳

第一轮 WP 结果是：

```text
Fail_6_8
```

失败的 proof obligation 是：

```text
typed_main_loop_invariant_2_preserved
```

trace 中的 `selected_po.json` 显示，该 preservation goal 最后变成：

```text
Assume {
  When: 7 <= j.
  Have: 6 <= j.
}
Prove: false.
```

`Prove: false` 通常说明当前 invariant 集合在循环回边处产生了矛盾，或者包含了无法 preservation 的错误规格。诊断结果把它归类为：

```text
category = wrong_spec
target_kind = invariant_preservation
recommended_action = rewrite_annotation
```

baseline3 首先尝试把：

```c
loop invariant j < i ==> (j == 6 && i == 11);
```

改成：

```c
loop invariant j < i ==> (i + j == 17);
```

这一步仍然没有真正抓住程序的 inductive relation，因此下一轮继续失败。

##### round 01：找到真正的核心关系 `i + 2*j == 21`

第二轮 WP 结果是：

```text
Fail_5_8
```

失败仍然集中在 invariant preservation。baseline3 进一步判断，原来的分支式 invariant：

```c
loop invariant j >= i ==> (i + j == 11);
loop invariant j < i ==> (i + j == 17);
```

不适合作为循环不变式。于是它把整个 loop annotation 重写为：

```c
/*@
  loop invariant i + 2*j == 21;
  loop assigns i, j;
  loop variant j;
*/
```

这是本案例最关键的一步。因为：

```text
初始状态: i + 2*j = 1 + 2*10 = 21
循环更新: i' = i + 2, j' = j - 1
更新后:   i' + 2*j' = (i + 2) + 2*(j - 1) = i + 2*j
```

所以：

```text
i + 2*j == 21
```

是真正 preserved 的线性关系。

##### round 02：核心关系还不足以证明 assertion

第三轮 WP 结果是：

```text
Fail_5_6
```

此时 invariant preservation 已经基本解决，剩下失败点变成最后的 assertion：

```text
Prove: j = 6
```

trace 中的 failed goal 是：

```text
Assume {
  Type: is_sint32(j) /\ is_sint32(21 - (2 * j)).
  Have: j <= 6.
}
Prove: j = 6.
```

这说明 WP 当前只能推出：

```text
j <= 6
```

但还缺少能推出 `j >= 6` 的信息。因此 baseline3 判断当前规格是 missing/weak spec，于是添加一些基础 bounds：

```c
loop invariant i >= 1;
loop invariant j <= 10;
loop invariant i <= 21;
loop invariant 0 <= j <= 10;
```

这些 bounds 都是正确的，但它们仍然不能单独推出 `j == 6`。

##### round 03：尝试添加桥接 invariant `i <= j + 2`

第四轮 WP 结果是：

```text
Fail_13_14
```

失败点仍然是：

```text
typed_main_assert
```

此时 WP 在 assertion 处已经知道：

```text
j <= 10
0 <= j
j <= 6
```

但还是无法证明：

```text
j == 6
```

baseline3 的 missing_spec repair 于是添加了一个关系型 invariant：

```c
loop invariant i <= j + 2;
```

LLM 给出的理由是：循环退出时有 `j < i`，如果再有 `i <= j + 2`，就能把 `i` 和 `j` 的距离限制住，再结合：

```text
i + 2*j == 21
```

尝试推出 `j == 6`。

这一步的方向是对的：它意识到仅有 `j <= 6` 不够，需要一个额外关系来补出下界。但是 `i <= j + 2` 太强了，不是 preserved 的 loop invariant。

##### round 04：发现 `i <= j + 2` 过强，并弱化为 `i <= j + 5`

第五轮 WP 结果是：

```text
Fail_15_16
```

失败点变成新加入 invariant 的 preservation：

```text
typed_main_loop_invariant_6_preserved
```

WP 要证明的目标是：

```text
Prove: 8 <= j.
```

但当前 assumptions 只有：

```text
7 <= j
j <= 10
0 <= j
0 < j
```

显然 `7 <= j` 不能推出 `8 <= j`，所以 `i <= j + 2` 不可保持。直观地看，循环体执行：

```text
i := i + 2
j := j - 1
```

会让 `i - j` 每轮增加 3。因此如果 invariant 把 `i` 和 `j` 限得太近，就容易在循环回边失败。

baseline3 因此把：

```c
loop invariant i <= j + 2;
```

弱化为：

```c
loop invariant i <= j + 5;
```

这个关系更弱，所以更容易被 preserved。同时它仍然足够帮助最终 assertion。

##### round 05：全部 WP 目标通过

第六轮 WP 结果是：

```text
Pass_16_16
```

最终保存的成功文件中，loop annotation 是：

```c
/*@
  loop invariant i + 2*j == 21;
  loop invariant i >= 1;
  loop invariant j <= 10;
  loop invariant i <= 21;
  loop invariant 0 <= j <= 10;
  loop invariant i <= j + 5;
  loop assigns i, j;
  loop variant j;
*/
```

#### 6.3 最终为什么可以证明 `j == 6`

最终证明依赖两个关键方向。

第一，循环退出条件给出：

```text
not(j >= i)
```

也就是：

```text
j < i
```

对整数来说，可以推出：

```text
j + 1 <= i
```

再结合核心 invariant：

```text
i + 2*j == 21
```

可以推出：

```text
j <= 6
```

第二，最终加入的关系：

```text
i <= j + 5
```

结合：

```text
i + 2*j == 21
```

可以推出：

```text
21 - 2*j <= j + 5
16 <= 3*j
```

由于 `j` 是整数，所以得到：

```text
6 <= j
```

于是有：

```text
j <= 6 /\ 6 <= j  =>  j == 6
```

这解释了为什么最终版本能够证明最后的 assertion。

#### 6.4 和 symbolic abduction 的关系

这个 trace 和后面的 symbolic abduction 实验是对应的。

在 round 02 和 round 03 中，WP failed goal 的本质是：

```text
Assume:
  j <= 6

Prove:
  j == 6
```

这和 symbolic abduction 在 `wptest.txt` 中看到的问题一致：当前证明已经有上界 `j <= 6`，但缺少下界 `6 <= j`。

symbolic abduction 对同类 failed goal 输出的第一个候选正是：

```text
6 <= j
```

这说明两者解决的是同一个证明缺口：

```text
baseline3 trace:
  通过多轮 LLM repair，最终加入 i <= j + 5，
  再间接推出 6 <= j。

symbolic abduction:
  直接从 failed goal 层面指出，
  当前目标缺少的桥接事实是 6 <= j。
```

因此，svcomp/235 这个案例很适合说明 baseline3 和 symbolic abduction 的互补关系：baseline3 能把错误 invariant 一步步修成可验证的 inductive invariant；symbolic abduction 则能在 assertion failed goal 上更直接地指出当前证明缺少什么中间事实。

## 二、symbolic abduction 原型实验

### 1. 实现目标

symbolic abduction 原型位于：

```text
abduction/symbolic_abduction_v1/
```

它目前不接入 baseline3 pipeline，是一个隔离实验模块。

目标是复现 GPiD-style abduction 的核心逻辑：

```text
给定 failed WP goal:
  assumptions A
  goal G

寻找候选事实 H，使得:
  A /\ H => G
```

等价 SMT 检查：

```text
A /\ H /\ not(G)  UNSAT
```

如果 UNSAT，则说明 `H` 是一个 sufficient abducible：补上它之后，当前失败目标可以被推出。

### 2. 代码结构

```text
abduction/symbolic_abduction_v1/
├── README.md
├── __init__.py
├── linear.py
├── wp_goal.py
├── smt.py
├── backend.py
└── debug_wp.py
```

各文件职责：

```text
linear.py
  定义 LinExpr 和 Atom。
  支持解析简单整数线性表达式和比较谓词，例如 i + 1 <= j。

wp_goal.py
  从 wp.txt 中提取第一个 Failed/Timeout/Unknown 的 Goal block。
  解析 Assume 中的 Have 行和 Prove 目标。

smt.py
  封装 Z3 CLI。
  生成临时 .smt2 文件并调用 /opt/z3-4.7.1/bin/z3。

backend.py
  abduction 主逻辑。
  生成候选 H，检查 consistent、already_known、sufficient、restates_goal。

debug_wp.py
  命令行入口。
  输入 wp.txt，输出 JSON 格式候选事实。
```

### 3. 当前支持的公式范围

当前版本只支持简单 QF_LIA 片段。

支持：

```text
i
j
10
i + 1
j - 2
i <= j
i < j
i == j
i + 1 <= j
0 <= i
j <= 10
```

暂不支持：

```text
array
pointer
quantifier
function call
nonlinear arithmetic
复杂 ACSL predicate
完整 C/ACSL AST
```

不支持的 `Have:` predicate 会被跳过，不做近似翻译，以避免引入错误语义。

### 4. 单 literal abduction

对每个候选 `H`，当前后端会做三个核心检查。

consistent：

```text
A /\ H  SAT ?
```

如果不可满足，说明 `H` 和 assumptions 冲突，直接丢弃。

already_known：

```text
A /\ not(H)  UNSAT ?
```

如果不可满足，说明 `A => H`，即 `H` 已经是已知事实。

sufficient：

```text
A /\ H /\ not(G)  UNSAT ?
```

如果不可满足，说明 `A /\ H => G`，该候选可以解释当前证明缺口。

此外还会标记 `restates_goal`。如果 `H` 和 `G` 完全相同，则说明它只是重述目标，虽然 sufficient，但不一定有研究价值。

### 5. Multi-literal abduction

当前已经实现 bounded multi-literal 搜索，默认：

```text
--max-size 2
```

也就是不仅搜索：

```text
H
```

还搜索：

```text
H1 /\ H2
```

流程是：

```text
1. 先检查所有单 literal。
2. 如果单 literal 已经 sufficient，记录为最小解释。
3. 如果单 literal consistent、不是 already_known、但单独不 sufficient，则进入 combo_pool。
4. 对 combo_pool 中的 top-N 候选做组合搜索。
5. 检查 A /\ H1 /\ H2 /\ not(G) 是否 UNSAT。
6. 如果某个组合包含已经 sufficient 的子集，则跳过，避免输出无意义超集。
```

`--max-combo-candidates` 用于控制组合爆炸。当前默认保留 40 个候选进入组合阶段。

### 6. 实验运行方式

以 svcomp/235 的 WP 输出为例：

```bash
python3 -m abduction.symbolic_abduction_v1.debug_wp \
  results/baseline1_llm_direct/failed_cases/svcomp/235/wp.txt \
  --visible i,j \
  --max-size 2 \
  --max-combo-candidates 20 \
  --max-results 10
```

在 svcomp/235 的一个失败目标上，当前工具会输出类似：

```text
i + 1 <= j
i + 1 < j
i + 2 <= j
i + 2 < j
...
```

这些候选都满足：

```text
A /\ H => i < j
```

其中 `i + 1 <= j` 是一个比直接重述 `i < j` 更有意义的桥接事实。

### 7. symbolic abduction 的意义

baseline3 目前依赖 LLM 从 WP failed goal 中猜测中间事实。symbolic abduction 原型把这一步部分符号化：

```text
LLM 猜 H
  -> 枚举 H
  -> SMT 检查 A /\ H => G
```

这有几个好处：

1. 候选事实可验证  
   每个输出的 `H` 都经过 Z3 检查，保证它确实足以推出当前目标。

2. 可以过滤明显坏候选  
   矛盾候选、已有事实、直接重述目标都可以被标记或过滤。

3. 可以生成多 literal 解释  
   有些目标不能由单个事实推出，但可以由 `H1 /\ H2` 推出，例如 `i <= j /\ j <= i => i == j`。

4. 可以作为 baseline3 的候选生成器  
   后续可以用 symbolic abduction 替换或辅助 baseline3 的 LLM assertion candidate prompt。

### 8. 一个具体案例

本节使用 `abduction/symbolic_abduction_v1/test.c` 和对应的 WP 输出
`abduction/symbolic_abduction_v1/wptest.txt` 作为一个具体案例，说明 symbolic abduction
当前能够做什么。

#### 8.1 程序背景

`test.c` 中的核心程序来自 svcomp/235，主要逻辑如下：

```c
i = 1;
j = 10;

while (j >= i) {
    i = i + 2;
    j = j - 1;
}

//@ assert(j == 6);
```

这个循环每次让 `i` 增加 2，让 `j` 减少 1。程序希望在循环结束后证明：

```text
j == 6
```

当前 `test.c` 里已经包含了很多由 LLM 或前序流程生成的 loop invariant。WP 的结果显示，大部分目标已经能够证明：

```text
[wp] 58 goals scheduled
[wp] Proved goals:   57 / 58
Timeout:             1
```

也就是说，这个例子不是完全没有信息，而是只剩最后一个 assertion goal 没有被证明。

#### 8.2 WP 失败目标

在 `wptest.txt` 里，失败目标是最后的 assertion：

```text
Goal Assertion (... line 47):
Assume {
  Have: 5 <= j.
  Have: 0 < j.
  Have: j <= 10.
  Have: (j + (x_1 / 2)) = 10.
  Have: (x_2 % 2) = 1.
  Have: 2 <= j.
  Have: j <= 6.
  Have: (x_1 % 2) = 0.
}
Prove: j = 6.
Prover Z3 4.8.12 returns Timeout
Prover Alt-Ergo returns Timeout
```

从这个 failed goal 可以看出，WP 在证明 `j = 6` 时已经知道：

```text
j <= 6
```

但它没有一个足够强的下界。已有的下界包括：

```text
5 <= j
2 <= j
0 < j
```

这些事实只能推出 `j` 至少是 5、2 或 1，不能推出 `j` 必须等于 6。要证明 `j = 6`，最直接缺失的是：

```text
6 <= j
```

因为：

```text
j <= 6 /\ 6 <= j  =>  j == 6
```

这正是 symbolic abduction 要找的桥接事实。

#### 8.3 运行 symbolic abduction

对这个 WP 输出运行当前原型：

```bash
python3 -m abduction.symbolic_abduction_v1.debug_wp \
  abduction/symbolic_abduction_v1/wptest.txt \
  --max-size 1 \
  --max-results 5
```

输出结果为：

```json
[
  {
    "predicate": "6 <= j",
    "sufficient": true,
    "consistent": true,
    "already_known": false,
    "restates_goal": false,
    "size": 1,
    "score": 11.5,
    "reason": "assumptions and this abducible make the negated goal unsatisfiable"
  },
  {
    "predicate": "j == 6",
    "sufficient": true,
    "consistent": true,
    "already_known": false,
    "restates_goal": true,
    "size": 1,
    "score": 8.5,
    "reason": "assumptions and this abducible make the negated goal unsatisfiable; this fact contains the goal and should be treated cautiously"
  }
]
```

#### 8.4 结果解释

第一个结果是：

```text
6 <= j
```

它的几个标记含义如下：

```text
sufficient = true
```

表示在当前 WP assumptions `A` 下，加入 `6 <= j` 以后，目标 `j == 6` 可以被推出。也就是：

```text
A /\ (6 <= j) => (j == 6)
```

等价地，SMT 检查的是：

```text
A /\ (6 <= j) /\ not(j == 6)
```

该公式不可满足，所以 `6 <= j` 是一个 sufficient abducible。

```text
consistent = true
```

表示 `6 <= j` 不会和当前 assumptions 冲突。因为当前 assumptions 里已经有 `j <= 6`，加入 `6 <= j` 后得到的是 `j == 6`，不是矛盾。

```text
already_known = false
```

表示当前 assumptions 本身还不能推出 `6 <= j`。换句话说，这不是 WP 已经显式知道的事实，而确实是一个缺失的桥接事实。

```text
restates_goal = false
```

表示它不是简单把目标 `j == 6` 原样加回去。相比直接补一句 `j == 6`，`6 <= j` 更像是一个有解释价值的中间事实：当前证明已经有上界 `j <= 6`，缺的是与之配对的下界。

第二个结果是：

```text
j == 6
```

它当然也能推出目标，所以 `sufficient = true`。但是它被标记为：

```text
restates_goal = true
```

这说明它只是直接重述证明目标。这样的候选在逻辑上有效，但对修复程序帮助较小，因为它没有告诉我们为什么 `j == 6` 应该成立。当前排序也把 `j == 6` 放在 `6 <= j` 后面，说明原型更偏好非重述目标的桥接事实。

#### 8.5 这个案例说明了什么

这个例子说明 symbolic abduction 原型已经可以在一个真实 WP failed goal 中完成以下事情：

1. 从 WP 输出中定位失败目标  
   该例子里定位到的是最后的 assertion goal：`Prove: j = 6`。

2. 提取已有 assumptions  
   例如 `5 <= j`、`j <= 6`、`2 <= j` 等。

3. 枚举候选 abducible  
   当前运行参数是 `--max-size 1`，所以只搜索单个 literal 的候选。

4. 用 SMT 判断候选是否足以推出目标  
   对每个候选检查 `A /\ H /\ not(G)` 是否不可满足。

5. 区分有意义的桥接事实和目标重述  
   `6 <= j` 是更有价值的候选，`j == 6` 被标记为目标重述。

从 baseline3 的角度看，这个案例对应的正是 assertion probe 想解决的问题：当 WP 在某个 goal 上失败时，系统需要猜一个中间事实来帮助证明。baseline3 目前主要依赖 LLM 从 failed goal 中生成候选 assertion，而 symbolic abduction 可以把这一步变成更可控的符号枚举和 SMT 验证。

在这个例子中，LLM 可能需要从上下文里猜出：

```text
//@ assert 6 <= j;
```

而 symbolic abduction 可以直接从失败目标中推出：如果补充 `6 <= j`，那么当前目标 `j == 6` 就能被证明。

#### 8.6 需要注意的限制

当前原型证明的是局部充分性：

```text
A /\ H => G
```

也就是说，它能说明 `H` 对当前 failed goal 有帮助，但还没有完整检查 `H` 是否适合作为 loop invariant，尤其还没有检查：

```text
1. H 在循环入口是否成立。
2. H 在循环体执行后是否 preserved。
3. H 加入原程序后是否会引入新的 WP 失败目标。
```

因此，当前 symbolic abduction 更准确地说是一个“失败目标桥接事实生成器”，还不是完整的 invariant synthesizer。

不过在这个具体案例中，`6 <= j` 从程序语义上也是合理的。初始时 `j = 10`，循环过程中 `j` 逐步下降；循环结束时，已有信息给出了 `j <= 6`，而缺失的 `6 <= j` 正好补足了证明 `j == 6` 所需的另一半。后续如果加入 loop preservation 检查，就可以进一步判断这个候选是否能作为真正可保留的 loop invariant 或中间 assertion 使用。

### 9. 小结

这个 abduction 原型目前已经验证了一个可行方向：对于 WP failed goal，不一定完全依赖 LLM 猜测中间断言，而是可以先把失败目标形式化为：

```text
A => G
```

然后自动寻找：

```text
H，使得 A /\ H => G
```

在 `test.c` / `wptest.txt` 这个案例中，工具成功从失败目标 `j = 6` 中找到了非目标重述的桥接事实 `6 <= j`。这说明 symbolic abduction 可以作为 baseline3 的一个候选生成模块，用来提高中间 assertion 的可解释性和可验证性。
